# Celebration File Lookup

This document describes how the package resolves a celebration code to a YAML file on disk.

## Directory structure

All liturgical data files live under `assets/calendar_data/`:

```
calendar_data/
  ferial_days/          # Ordinary ferial days: ot_3_5.yaml, advent_2_3.yaml, etc.
  sanctoral/
    roman/              # Universal Roman calendar feasts and solemnities
    belgium/
    france/
    lyon/
    ...                 # Other geographic locations
  commons/              # Hierarchical commons (apostles, martyrs, etc.)
  index.json            # Code → directory lookup table
```

## Celebration codes

A celebration code is the string stored in the calendar's `feastList` or as a `defaultCelebrationTitle`. It encodes both the origin (location) and the feast name:

| Code | File |
|---|---|
| `roman/corpus_domini` | `sanctoral/roman/corpus_domini.yaml` |
| `roman/nativity` | `sanctoral/roman/nativity.yaml` |
| `lyon/saint_pothinus_bishop` | `sanctoral/lyon/saint_pothinus_bishop.yaml` |
| `ot_3_5` | `ferial_days/ot_3_5.yaml` |
| `advent_2_3` | `ferial_days/advent_2_3.yaml` |

The `roman/` prefix indicates the universal Roman calendar. Geographic prefixes (`lyon/`, `france/`, `belgium/`, etc.) indicate a local sanctoral. Ferial day codes have no prefix.

## Path construction

`office_detection.dart` constructs the file path as:

```dart
final filePath = ferialDayCheck(c.code) ? ferialFilePath : sanctoralFilePath;
// Result: 'calendar_data/ferial_days' or 'calendar_data/sanctoral'

dataLoader.loadYaml('$filePath/${c.code}.yaml');
// e.g. 'calendar_data/sanctoral/roman/corpus_domini.yaml'
//      'calendar_data/ferial_days/ot_3_5.yaml'
```

`ferialDayCheck()` returns true for codes matching the pattern `season_week_day` (e.g. `ot_3_5`, `advent_2_3`, `lent_4_6`, `easter_2_1`).

## How codes enter the calendar

Codes are assigned in two ways:

### 1. `defaultCelebrationTitle` (root day)

Set directly in `main_calendar_fill.dart` for structurally distinct days:

```dart
defaultCelebrationTitle: 'roman/nativity'       // Christmas Day
defaultCelebrationTitle: 'roman/epiphany'        // Epiphany
defaultCelebrationTitle: 'roman/baptism'         // Baptism of the Lord
defaultCelebrationTitle: 'roman/ascension'       // Ascension
defaultCelebrationTitle: 'roman/pentecost'       // Pentecost
defaultCelebrationTitle: 'roman/christ_king'     // Christ the King
defaultCelebrationTitle: 'roman/mary_mother_of_god'
defaultCelebrationTitle: 'roman/holy_family'
defaultCelebrationTitle: 'roman/christmas_${date.day}'  // Christmas Octave (26–31 Dec)
defaultCelebrationTitle: 'ot_3_5'               // Ordinary ferial day
```

For all other days (Ordinary Time, Advent, Lent, Eastertide), the `defaultCelebrationTitle` is a ferial code and the day's feast(s) are added separately via `feastList`.

### 2. `feastList` entries

Added via `calendar.addItemToDay()`. These are the optional celebrations on a given day, sorted by precedence:

```dart
// In main_calendar_fill.dart — universal solemnities
calendar.addItemToDay(date, 3, 'roman/corpus_domini');
calendar.addItemToDay(date, 3, 'roman/sacred_heart');
calendar.addItemToDay(date, 3, 'roman/all_saints');
// etc.

// In location_loader.dart — local feasts
calendar.addItemToDay(date, precedence, 'roman/${feast.key}');
calendar.addItemToDay(date, precedence, 'lyon/${feast.key}');
```

### 3. Hardcoded in `office_detection.dart`

The Marian memory (optional memorial on ordinary Saturdays) is injected directly:

```dart
allCelebrations.add((code: 'roman/virgin-mary-memory', precedence: 12, ...));
```

## Ferial resolution — privileged days

The ferial resolution files (`ferial_morning_resolution.dart`, etc.) handle days that are ferial in structure but have dedicated content. They build paths directly without going through the code-based lookup:

| Days | Path pattern |
|---|---|
| Privileged Advent days (17–24 Dec) | `sanctoral/roman/advent_17.yaml` … `advent_24.yaml` |
| Christmas Octave (26–31 Dec) | `sanctoral/roman/christmas_26.yaml` … `christmas_31.yaml` |
| Days before Epiphany (2–7 Jan) | `sanctoral/roman/christmas-ferial_before_epiphany_2.yaml` … |

## `index.json`

Generated by `scripts/make_index.py` by scanning `sanctoral/` recursively. Maps each celebration code to its directory and metadata (title, color, commons).

The key in `index.json` is the code as it appears in the calendar (e.g. `roman/corpus_domini`), so a direct lookup `index[celebrationCode]` resolves immediately.

`index.json` has two roles at runtime:

1. **Title/metadata display** — used by the Flutter consumer app (`celebration_index.dart`) to resolve a code to its title and commons.
2. **Key resolution during calendar build** — `LiturgyData` loads the key set from `index.json` on startup and passes it as `knownCodes: Set<String>` down to `addItemToDay`. When a location declares a feast that already exists in the calendar under a different prefix (e.g. `france/louis_ix_of_france` vs `roman/louis_ix_of_france`), `addItemToDay` checks `knownCodes` to decide whether the new key has its own YAML file. If not, it preserves the existing qualified key and only updates the precedence (see [Key-prefix semantics](#key-prefix-semantics) below).

## Calendar date API

All feast entries in the calendar are managed through `Calendar` methods in `calendar_class.dart`. These are the functions that place, move, or remove codes from `DayContent.feastList`.

### Adding

| Method | Description |
|---|---|
| `addItemToDay(date, precedence, key, {knownCodes})` | Add or update a feast. Deduplicates by base name (strips prefix). Returns the key actually stored. See [Key-prefix semantics](#key-prefix-semantics) for collision behaviour. |
| `addItemRelatedToFeast(date, shift, prec, key, {knownCodes})` | Shorthand for `addItemToDay(date.shift(shift), prec, key)`. Used for feasts relative to a mobile date (e.g. Notre-Dame de Fourvière = Saturday after 2nd Sunday of Easter). Returns the stored key. |
| `addFeastsToCalendar(feasts, year, mainFeasts)` | Bulk add from a `Map<String, FeastDates>`. Handles liturgical-year boundary (tries current year, falls back to previous calendar year if date is past Christ the King). |

### Moving

| Method | Description |
|---|---|
| `moveItemToDate(feastName, newDate, precedence)` | Move a feast to an absolute date. Searches by base name (prefix-agnostic), so `peter_canisius_priest` finds `roman/peter_canisius_priest` in the calendar. Preserves the qualified key when re-adding. Returns the qualified key found, or `null` if not present. |

This is the method called by `Location.applyToCalendar()` for the `move:` section of location YAML files. Because the search is prefix-agnostic, the `move:` section can relocate feasts from any origin — `roman/`, parent-location, or the location's own `feasts:` section.

### Key-prefix semantics

`addItemToDay` and `moveItemToDate` both use `_baseName()` internally to strip the directory prefix before comparing. This means:

- `roman/peter_canisius_priest` and `peter_canisius_priest` are considered the same feast for dedup/search purposes.
- When a more specific key replaces a less specific one at the same precedence, the new key wins (locality override).

**Precedence-only override (no own YAML file)**

A location can change the precedence of a Roman feast without providing its own YAML content. For example, `france.yaml` declares `louis_ix_of_france` with `precedence: 10` to elevate the feast from the Roman value of 12 — but there is no `sanctoral/france/louis_ix_of_france.yaml`.

When `addItemToDay` is called with `france/louis_ix_of_france` and `knownCodes` is provided, it checks whether `france/louis_ix_of_france` is in the index. Since it is not, it keeps `roman/louis_ix_of_france` as the stored key and only moves the entry to the new precedence bucket. The YAML file path therefore remains valid.

If `knownCodes` is omitted (all callers outside `Location.applyToCalendar`), the old behaviour applies: the new key always replaces the existing one.
